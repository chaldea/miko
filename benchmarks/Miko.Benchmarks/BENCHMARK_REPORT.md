# Miko 渲染引擎基准测试报告

**环境**: AMD Ryzen 7 8700G w/ Radeon 780M Graphics, .NET 10.0.7, Windows 11, BenchmarkDotNet v0.14.0

---

## 每帧时间（Full Frame Pipeline）

完整帧 = 样式解析 + 布局 + 渲染，一次性测量：

| 场景 | 每帧耗时 | 内存分配 | 60fps 预算占比 |
|------|---------|---------|--------------|
| 真实页面 ~90 元素（全量渲染） | **1.58 ms** | 923 KB | 9.5% |
| 大页面 500 元素（全量渲染） | **4.83 ms** | 3,143 KB | 29% |
| 真实页面（增量，1个脏区域） | **0.91 ms** | 708 KB | 5.5% |
| 大页面（增量，1个脏区域） | **1.81 ms** | 2,039 KB | 10.9% |

结论：真实页面场景下每帧 1.6ms，60fps（16.6ms）预算内非常充裕。增量渲染可以节省约 42% 的帧时间。

注意：帧时间中布局占了大头（因为当前每帧都重新 Layout），如果能跳过未变化元素的布局，增量帧可以更快。

## 局部重绘 vs 全量重绘 — 拐点分析

500 个元素的树，不同脏区域数量下增量渲染与全量渲染的对比：

| 脏区域数 | 增量渲染耗时 | 相对全量渲染 | 结论 |
|---------|------------|------------|------|
| 1 | 302 μs | 11% | 快 9 倍 |
| 2 | 457 μs | 16% | 快 6 倍 |
| 3 | 565 μs | 19% | 快 5 倍 |
| 5 | 631 μs | 17% | 快 6 倍 |
| 8 | 885 μs | 26% | 快 4 倍 |
| 10 | 946 μs | 24% | 快 4 倍 |
| 15 | 1,586 μs | 42% | 快 2.4 倍 |
| 20 | 2,372 μs | 66% | 快 1.5 倍 |
| **30** | **3,994 μs** | **110%** | **比全量慢 10%** |
| 50 | 7,034 μs | 207% | 比全量慢 2 倍 |

**拐点在 20-30 个脏区域之间**。当脏区域超过 ~25 个时，增量渲染反而比全量渲染更慢。建议在 `RenderEngine` 中加入阈值判断：脏区域超过 20 个时直接回退到全量渲染。

## 各模块详细基准

### 0. Table 表格渲染

大规模表格性能测试（表格结构：table > thead > tbody > tr > td，使用 Flex 布局行）：

| 测试 | 耗时 | 内存分配 | GC (Gen0/Gen1) |
|------|------|---------|---------------|
| Layout 1万行 x 5列 (5万单元格) | 263 ms | 216 MB | 27K / 9K |
| Layout 10万行 x 5列 (50万单元格) | 2.67 s | 2.25 GB | 282K / 78K |
| Layout 100万行 x 5列 (500万单元格) | **30.0 s** | **23.4 GB** | 2.9M / 771K |
| Render 1万行 x 5列 | 540 ms | 319 MB | 39K / 9K |
| Render 10万行 x 5列 | 5.41 s | 3.36 GB | 421K / 78K |

分析：
- **线性扩展性良好**：1万→10万行耗时增长 10 倍，符合 O(n) 复杂度
- **100万行瓶颈**：30 秒布局 + 23GB 内存，不适合一次性渲染全部数据
- **渲染开销**：渲染耗时约为布局的 2 倍（1万行：263ms 布局 vs 540ms 全流程）
- **GC 压力**：100万行触发 2.9M 次 Gen0 GC，内存分配是主要瓶颈

建议：
- 大表格必须使用虚拟滚动（只渲染可见行）
- 10万行以内可接受（2.7秒布局），但仍建议分页或虚拟化
- 1万行以下体验流畅（263ms 布局，540ms 全流程）

### 1. Layout 布局引擎

| 测试 | 耗时 | 内存分配 |
|------|------|---------|
| Block 10个子元素 | 26 μs | 39 KB |
| Block 1000个子元素 | 3,223 μs | 4,061 KB |
| Block 50层嵌套 | 19 μs | 109 KB |
| Flex 10个子元素 | 4 μs | 24 KB |
| Flex 200个子元素 | 77 μs | 445 KB |
| Inline 100个元素 | 550 μs | 574 KB |
| 混合真实页面 | 779 μs | 691 KB |

分析：Flex 布局效率最高（线性扩展良好）。Inline 布局相对较慢（100个元素就需要 550μs），可能是文本测量开销。Block 大规模布局（1000个）的 4MB 内存分配值得关注。

### 2. Style 样式解析

| 测试 | 耗时 | 内存分配 |
|------|------|---------|
| 5条规则/10元素 | 28 μs | 39 KB |
| 100条规则/100元素 | 262 μs | 681 KB |
| 150条复杂选择器/100元素 | 658 μs | 1,116 KB |
| 100条规则/深层继承 | 86 μs | 209 KB |

分析：样式解析是 O(n×m) 复杂度（元素数×规则数），复杂选择器场景下内存分配超过 1MB。这是大型应用的主要瓶颈之一。

### 3. Render 渲染引擎

| 测试 | 耗时 | 内存分配 |
|------|------|---------|
| 全量渲染 10个元素 | 95 μs | 19 KB |
| 全量渲染 500个元素 | 2,113 μs | 1,121 KB |
| 增量渲染 1个脏区域 | 70 μs | 18 KB |
| 增量渲染 20个脏区域 | 2,381 μs | 1,116 KB |
| 含文本渲染（真实页面） | 490 μs | 231 KB |

分析：增量渲染单区域（70μs）比全量渲染（2,113μs）快 30 倍，脏区域优化有效。但 20 个脏区域时性能退化到与全量渲染持平。

### 4. DirtyRegion 脏区域管理

| 测试 | 耗时 | 内存分配 |
|------|------|---------|
| 50个不重叠区域 | 12.7 μs | 0 |
| 50个重叠区域 | 12.7 μs | 0 |
| 50个相邻区域 | 11.7 μs | 0 |

分析：脏区域管理非常轻量，零堆分配，三种场景性能一致（~12μs/50个区域）。不是性能瓶颈。

### 5. TreeTraversal DOM树遍历

| 测试 | 耗时 | 内存分配 |
|------|------|---------|
| 前序遍历 1000节点 | 2.6 μs | 88 B |
| 后序遍历 1000节点 | 1.9 μs | 88 B |
| 层序遍历 1000节点 | 6.3 μs | 33 KB |
| FindById 1000节点 | 4.3 μs | 0 |
| FindByClass 1000节点 | 32.7 μs | 123 KB |

分析：前序/后序遍历几乎零分配，非常高效。`FindByClass` 分配 123KB 是因为递归创建 List 并 AddRange，可优化为单次遍历收集。层序遍历的 33KB 来自 Queue 分配。

## 主要优化建议

1. **增量渲染阈值** — 脏区域超过 20 个时应回退到全量渲染，避免 N 次全树遍历的退化
2. **Inline 布局** — 100个元素 550μs，是 Flex 200个元素的 7 倍，文本测量是瓶颈
3. **样式解析内存** — 复杂选择器场景 1MB+ 分配，可考虑选择器索引或缓存
4. **FindByClass** — 递归 List 合并导致 123KB 分配，可优化为传入共享 List
5. **布局缓存** — 当前每帧都重新 Layout，可对未变化的子树跳过布局计算

```bash

dotnet run -c Release --project benchmarks/Miko.Benchmarks -- --filter "*" --job short

# 仅运行帧管线和拐点测试
dotnet run -c Release --project benchmarks/Miko.Benchmarks -- --filter "*FramePipeline*" "*TippingPoint*" --job short

# 运行特定类
dotnet run -c Release --project benchmarks/Miko.Benchmarks -- --filter "*LayoutBenchmarks*"
```
